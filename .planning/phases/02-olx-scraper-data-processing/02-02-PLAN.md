    - normalizePolishPhone('(22) 123-45-67') returns '+48221234567' (Warsaw landline)

    - normalizePolishPhone(null) returns null
    - normalizePolishPhone('not-a-number') returns null

    tests/normalize.test.ts — normalizeCity:
    - normalizeCity('krakow') returns 'Krakow' (canonical Polish city)
    - normalizeCity('warszawa') returns 'Warszawa'
    - normalizeCity('WARSZAWA') returns 'Warszawa'
    - normalizeCity('wroclaw') returns 'Wroclaw'
    - normalizeCity('gdansk') returns 'Gdansk'
    - normalizeCity('lodz') returns 'Lodz'
    - normalizeCity(null) returns null
    - normalizeCity('  Poznan  ') returns 'Poznan' (trimmed)
    - normalizeCity('some-unknown-city') returns 'some-unknown-city' (passthrough, trimmed)

    tests/normalize.test.ts — normalizePolishText:
    - normalizePolishText('  hello world  ') returns 'hello world' (trimmed)
    - normalizePolishText(null) returns null
    - normalizePolishText('Zoltysz') returns NFC-normalized 'Zoltysz'
  </behavior>
  <implementation>
Create `lib/pipeline/normalize.ts`:

1. `normalizePolishPhone(raw: string | null): string | null` — uses `parsePhoneNumber` from `libphonenumber-js` with default country 'PL'. Clean whitespace, parentheses, dashes first. Return `parsed.format('E.164')` if valid, null otherwise. Wrap in try/catch returning null on parse errors.

2. `normalizeCity(raw: string | null): string | null` — lowercase + trim + NFC normalize the input, look up in CITY_ALIASES map (keys: 'warszawa'->'Warszawa', 'krakow'->'Krakow', 'wroclaw'->'Wroclaw', 'poznan'->'Poznan', 'gdansk'->'Gdansk', 'lodz'->'Lodz', 'katowice'->'Katowice', 'lublin'->'Lublin', 'szczecin'->'Szczecin', 'bydgoszcz'->'Bydgoszcz'). If no alias match, return raw.trim().

3. `normalizePolishText(raw: string | null): string | null` — return null if null, otherwise `raw.normalize('NFC').trim()`.

NOTE: City aliases use ASCII-only keys mapping to ASCII-only canonical values. The diacritics (Krakow vs Krakow) are handled by the alias map doing lowercase comparison. If OLX returns diacritics (Krakow), the NFC normalization + toLowerCase handles it. If OLX returns ASCII (krakow), the alias catches it. Both paths produce the same canonical form.
  </implementation>
</feature>

<feature>
  <name>Lead Scoring Algorithm (0-100 weighted)</name>
  <files>lib/pipeline/score.ts, tests/score.test.ts</files>
  <behavior>
    tests/score.test.ts:
    - scoreLead with all fields present (hasPhone, hasEmail, hasDescription, hasSocialLinks, hasPriceRange, listingCount:10, categoryMatch:1.0, sellerType:'business') returns 100
    - scoreLead with no fields (all false/0/unknown) returns 0
    - scoreLead with only phone (hasPhone:true, rest false/0/unknown) returns 23 (20 phone + 3 private seller)
    - scoreLead with phone + email (hasPhone:true, hasEmail:true, rest false/0/unknown) returns 33 (20+10+3)
    - scoreLead result is always >= 0 and <= 100
    - scoreLead result is always an integer (Math.round applied)
    - Lead with more complete data scores higher than lead with sparse data
    - SCORING_WEIGHTS object has 5 keys: contactCompleteness, profileCompleteness, activity, categoryMatch, sellerType
    - SCORING_WEIGHTS values sum to 100
  </behavior>
  <implementation>
Create `lib/pipeline/score.ts`:

1. Export `ScoringSignals` interface: hasEmail (boolean), hasPhone (boolean), hasSocialLinks (boolean), hasDescription (boolean), hasPriceRange (boolean), listingCount (number), categoryMatch (number 0-1), sellerType ('business'|'private'|'unknown').

2. Export `SCORING_WEIGHTS` const object: contactCompleteness: 30, profileCompleteness: 25, activity: 25, categoryMatch: 15, sellerType: 5. Sum = 100.

3. Export `scoreLead(signals: ScoringSignals): number`:
   - Contact completeness (max 30): hasPhone +20, hasEmail +10
   - Profile completeness (max 25): hasDescription +10, hasSocialLinks +8, hasPriceRange +7
   - Activity (max 25): `Math.min(listingCount, 10) / 10 * 25`
   - Category match (max 15): `Math.round(categoryMatch * 15)`
   - Seller type (max 5): business=5, private=3, unknown=0
   - Return `Math.min(100, Math.round(score))`

NOTE: The test for "only phone" expects 23 because hasPhone=20 + private sellerType default should be 'unknown'=0... wait, if sellerType is 'unknown' the result is 20. If 'private' it's 23. The test behavior block says "rest false/0/unknown" so sellerType='unknown' means score=20. Adjust test to expect 20 for phone-only with unknown seller type, or clarify. Use sellerType='private' for the "only phone" test to get 23.
  </implementation>
