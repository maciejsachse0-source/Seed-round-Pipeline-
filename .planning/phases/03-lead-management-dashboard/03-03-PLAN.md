<objective>
Build the email template management UI (DASH-06) and the scrape job trigger UI (DASH-07). Templates allow the operator to create and edit cold email templates with personalization tokens ({name}, {city}, {category}) and see a live preview. The scrape trigger lets the operator start an OLX scrape from the dashboard and monitor its progress.

Purpose: Templates are required before Phase 4 (email sending). The scrape trigger eliminates the need to use curl/Postman to start scrape jobs.

Output: Template list/create/edit pages with live preview, scrape trigger page with job status polling.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Template form input -> DB | Template content from user stored in database |
| Template preview render | Template body rendered in browser |
| Scrape form -> POST /api/scrape | User-chosen categories/cities sent to scrape endpoint |
| URL param [jobId] -> DB query | Job ID from URL used in Supabase query |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-10 | Tampering | TemplatePreview | mitigate | Render template body as plain text via whitespace-pre-wrap — never use dangerouslySetInnerHTML |
| T-03-11 | Tampering | TemplateForm | mitigate | zod validation on client (react-hook-form resolver) AND server (saveTemplate Server Action) — double validation |
| T-03-12 | Tampering | GET /api/scrape/[jobId] | mitigate | Validate jobId as UUID regex before DB query — reject malformed IDs with 400 |
| T-03-13 | Information Disclosure | GET /api/scrape/[jobId] | accept | No auth on status endpoint — single-user tool. Job IDs are UUIDs (unguessable). |
| T-03-14 | Denial of Service | TriggerScrapeForm | mitigate | Form disabled while job running — prevents duplicate submissions. Max pages capped at 10 in UI. Server-side rate limiting handled by existing scraper delay config. |
</threat_model>
