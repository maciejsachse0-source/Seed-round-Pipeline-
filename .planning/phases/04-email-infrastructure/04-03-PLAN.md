    - reply-check-worker queries email_events for the latest start_history_id
    - reply-check-worker calls pollForReplies and matches threadIds against email_events.gmail_thread_id
    - When a match is found: email_event.status -> 'replied', email_event.replied_at -> now, lead.status -> 'replied'
    - reply-check-worker stores the new historyId for next poll cycle

  <action>
1. Create `lib/email/reply-poller.ts`:
   - Export `pollForReplies(startHistoryId: string): Promise<{ replyThreadIds: string[], newHistoryId: string | undefined }>`
   - Create OAuth2 client using `google.auth.OAuth2` with GMAIL_CLIENT_ID, GMAIL_CLIENT_SECRET
   - Set credentials with GMAIL_REFRESH_TOKEN
   - Call `gmail.users.history.list({ userId: 'me', startHistoryId, historyTypes: ['messageAdded'], labelId: 'INBOX' })`
   - Extract threadIds from `res.data.history[].messagesAdded[].message.threadId`
   - Return unique threadIds and `res.data.historyId` as newHistoryId
   - Handle 404 response (historyId expired): return empty array and undefined newHistoryId — the worker should re-seed from the latest email_event

2. Create `lib/queue/workers/reply-check-worker.ts`:
   - Export `registerReplyCheckWorker()`:
     a. Get boss via `getBoss()`
     b. Schedule cron: `boss.schedule('email-reply-check', '*/15 * * * *', {})`
     c. Register worker: `boss.work('email-reply-check', async () => { ... })`
   - Inside worker handler:
     a. Query Supabase for the most recent email_event with non-null `start_history_id`, ordered by sent_at desc — this is the polling baseline
     b. If no events with start_history_id exist, log "no baseline" and return
     c. Call `pollForReplies(startHistoryId)`
     d. For each replyThreadId: query email_events where `gmail_thread_id = threadId` and `status = 'sent'`
     e. For each match: update email_event `status='replied'`, `replied_at=now()`
     f. For each match: load the lead, call `assertTransition(lead.status, LeadStatus.REPLIED)`, update lead status
     g. If newHistoryId returned: store it. Strategy: update the most recent email_event's start_history_id to the new value (this slides the polling window forward)
   - Log summary: "Reply check: found {n} replies in {m} threads"

3. Create `tests/email/reply-poller.test.ts`:
   - Mock googleapis with vi.mock('googleapis', ...)
   - Test: pollForReplies returns matching threadIds from history response
   - Test: pollForReplies returns empty array when no history
   - Test: pollForReplies handles 404 (expired historyId)
   - Test: reply-check-worker marks matched email_events as replied
   - Test: reply-check-worker transitions lead status to REPLIED

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-08 | Tampering | app/api/track/open/[eventId]/route.ts | mitigate | Validate eventId is UUID format before DB query; still return pixel on invalid ID (no information leak) |
| T-04-09 | Tampering | app/api/unsubscribe/route.ts | mitigate | HMAC verification via verifyUnsubscribeToken before any suppression action; reject tampered tokens with 400 |
| T-04-10 | Information Disclosure | app/api/unsubscribe/route.ts | mitigate | Same error message for missing params and invalid token ("Invalid unsubscribe link") — no enumeration possible |
| T-04-11 | Denial of Service | app/api/track/open/[eventId]/route.ts | accept | Fire-and-forget DB write means a flood of pixel requests creates DB load but does not block responses; acceptable at this scale |
| T-04-12 | Spoofing | lib/email/reply-poller.ts | mitigate | Gmail API calls use OAuth2 auth; refresh token stored server-side only |
</threat_model>
