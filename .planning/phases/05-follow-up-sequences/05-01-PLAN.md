## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Dashboard API -> sequence_config | PATCH endpoint accepts user input for follow-up config values |
| pg-boss job data -> follow-up worker | Job data contains leadId and sequenceStep from internal scheduling |
| follow-up worker -> sendColdEmail | Worker passes lead data to email send pipeline |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | /api/sequence-config PATCH | mitigate | Zod validation: max_follow_ups capped 0-10, interval_days capped 1-30 (Plan 02 Task 1) |
| T-05-02 | Denial of Service | sequence_config interval_days=0 | mitigate | Zod schema rejects interval_days < 1; follow-up.ts also guards sequenceStep >= 1 |
| T-05-03 | Information Disclosure | follow-up worker logs | accept | Logs contain leadId (UUID) and status only — no PII (email/name not logged) |
| T-05-04 | Elevation of Privilege | GDPR: follow-up to opted_out lead | mitigate | Worker checks lead.status before every send; opted_out and replied statuses stop the chain; suppression list checked inside sendColdEmail |
| T-05-05 | Repudiation | Sequence config changes not audited | accept | Single-user tool; updated_at timestamp on config row provides basic audit; full audit trail is v2 |
| T-05-06 | Tampering | pg-boss job data manipulation | accept | pg-boss stores jobs in PostgreSQL with RLS; only server-side code enqueues; no external job submission path |
</threat_model>
